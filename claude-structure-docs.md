## 1. What is the .claude/ System?

When you open Claude Code (Anthropic's CLI tool) inside this project, Claude reads the `.claude/`
directory to understand your specific codebase. Think of it as giving Claude a **project briefing**
before every session.

Without this system, Claude starts every session knowing nothing about your project — you'd have
to re-explain the architecture, conventions, and rules every time. With this system, Claude
walks in already knowing:

- The full medallion architecture (Bronze → Silver → Gold layers)
- Your naming conventions and layer contracts
- Existing bugs in the codebase
- How to run, test, debug, and commit your code
- Domain knowledge about PySpark, SQL Server, and data modeling

The system has six types of files, each with a distinct purpose:

| Type | Location | Purpose |
|---|---|---|
| `CLAUDE.md` | Project root | Auto-loaded every session — permanent project memory |
| `settings.json` | `.claude/` | Permissions (what Claude can run) + hook event wiring |
| Skills | `.claude/skills/` | Domain knowledge Claude consults when making decisions |
| Commands | `.claude/commands/` | Slash commands you type to trigger structured workflows |
| Agents | `.claude/agents/` | Specialist sub-brains for focused task areas |
| Hooks | `.claude/hooks/` | Python scripts that fire automatically on events |

## 3. CLAUDE.md — The Project Brain

**File location:** `CLAUDE.md` (project root)

**How it works:** Claude Code reads this file automatically at the start of every session.
You never need to tell Claude to read it — it always does. It is permanent project memory.

**What's inside:**

### Architecture Map
Every class name, file path, and their relationships. When you say "fix the gold layer",
Claude already knows that means `src/gold/aggregations.py` → `GoldLayer` class.

### Medallion Layer Contracts
The strict rules for each layer:
- **Bronze**: Zero business logic. Zero renaming. Faithful copy of source. Full overwrite mode.
- **Silver**: One `transform_<entity>()` method per table. Always snake_case aliases. PK null filter always last. Row-level only — no joins across tables.
- **Gold**: Always `left` join from the fact table (`order_items`). groupBy + agg only. Never inner joins (silently drops data).

**Example enforcement:** If you ask Claude "add a filter for only Electronics products in the Bronze layer", Claude will refuse because CLAUDE.md says Bronze must be a faithful copy of source. It will offer to add the filter in Silver instead.

### Known Bugs Section
Deliberately documents bugs that already exist:
- JDBC auth is broken (empty user/password strings — needs `integratedSecurity=true` in URL)
- JDBC driver jar not configured
- Unused StructType imports in Silver
- `tests/` directory is empty

**Why document bugs?** So Claude proactively flags them instead of copying the broken pattern when adding new features.

### What NOT to Do
Architectural anti-patterns Claude will refuse to write:
- `SparkSession` inside layer classes (only in `main.py`)
- `inner` join in Gold
- `select("*")` in Silver
- Hardcoded credentials

### Dev Runbook
All commands needed to run, test, validate, and debug the pipeline — available to Claude in every session.

## 4. settings.json — Permissions and Hooks

**File location:** `.claude/settings.json`

This file does two things: controls what Claude is allowed to run without prompting you,
and wires up which Python scripts fire on which events.

### Permissions

```json
"allow": [
  "Bash(python src/main.py)",   // run the pipeline — pre-approved, no popup
  "Bash(pytest *)",              // run tests — pre-approved
  "Bash(git add src/*)",         // stage source files only (not data/ or .env)
  ...
]
"deny": [
  "Bash(rm -rf *)",              // always blocked — no popup, just refused
  "Bash(git push --force *)",    // blocked
  "Bash(DROP DATABASE *)"        // blocked
]
```

**Key safety design:** `git add src/*` is allowed but `git add .` is not listed — so Claude can
only stage your source code, never accidentally capture `data/`, `logs/`, or `.env` files.

### Hook Events

Four events are wired:

| Event | Matcher | Script | What it does |
|---|---|---|---|
| `PreToolUse` | `Bash` | `pre_bash_guard.py` | Checks every bash command before running |
| `PostToolUse` | `Bash` | `post_bash_logger.py` | Logs every bash command after running |
| `PostToolUse` | `Write\|Edit\|MultiEdit` | `update_claude_docs.py` | Regenerates this doc if a .claude/ file changed |
| `Stop` | *(any)* | `session_summary.py` | Prints session summary when Claude Code closes |

## 5. Skills — Domain Knowledge

**File location:** `.claude/skills/*.md`

Skills are Markdown documents that give Claude deep, project-specific domain knowledge.
Each skill has a `description` frontmatter field that tells Claude when to consult it.

**How Claude uses them:** When you ask a question or request a task, Claude matches it
against skill descriptions. If the task involves PySpark DataFrame patterns, Claude reads
`pyspark-patterns.md`. If it involves adding a new table, Claude reads `data-modeling.md`.

### `medallion-architecture.md`
The foundational skill. Explains what medallion means IN THIS PROJECT specifically —
not the generic definition, but:
- Which tables live in which layer
- What the Silver contract is ("if a row is in Silver, its PK is valid")
- When to add a new Gold table vs. update an existing one
- The full checklist for adding a new domain table across all three layers

### `pyspark-patterns.md`
Contains the exact code patterns used in this codebase:
```python
# The exact JDBC read pattern for this project
df = spark.read.format("jdbc") \
    .option("url", jdbc_url) \
    .option("dbtable", "dbo.Products") \
    .option("driver", "com.microsoft.sqlserver.jdbc.SQLServerDriver") \
    .load()

# The Silver transform pattern every method must follow
df = bronze_df.select(
    F.col("ProductID").cast("int").alias("product_id"),  # rename + cast
    ...
)
df = df.filter(F.col("product_id").isNotNull())  # PK null filter ALWAYS last
```
Also includes anti-patterns: never use `toPandas()`, never use `inner` join in Gold,
never use `select("*")` in Silver.

### `data-quality.md`
PySpark code for DQ checks specific to this pipeline's data:
- Null rate per column
- Duplicate PK detection
- Referential integrity (orphaned FKs)
- Gold ↔ Silver revenue reconciliation
- Layer transition gates (don't run Silver if Bronze is empty)

### `sql-server-integration.md`
Documents the exact fix for the JDBC auth bug, the driver jar setup, and a failure
lookup table. When you hit `ClassNotFoundException: com.microsoft.sqlserver.jdbc.SQLServerDriver`,
Claude finds the exact fix by reading this skill instead of guessing.

### `git-versioning.md`
Branch strategy, commit message format (`feat(silver): add transform_inventory`), safe
staging rules, and which files are always safe vs. never safe to commit.

### `cicd-github-actions.md`
Why Java 11 must be installed BEFORE PySpark, why `SPARK_LOCAL_IP=127.0.0.1` is required
in CI, and a failure lookup table for common GitHub Actions + PySpark errors.

### `data-modeling.md`
The current star schema, why `order_items` is the fact table, why `line_total` belongs
in Silver not Gold, and the decision guide for whether a new table is a fact or dimension.

## 6. Slash Commands — Daily Workflows

**File location:** `.claude/commands/*.md`

Type `/command-name` in Claude Code. The filename (minus `.md`) is the command name.
Each command file pre-describes a multi-step workflow so you don't have to explain it.

### Command File Features

Three powerful syntax elements inside command files:

**1. `!`bash command`` — injects live context at invocation time**
```markdown
Current config state:
!`python -c "from src.config.config_manager import ConfigManager; c=ConfigManager(); print(c.get('sql_server.server'))"`
```
When you type `/run-pipeline`, this bash runs first and injects the output into Claude's context.
Claude sees your actual live config state before doing anything.

**2. `@filepath` — inlines file contents**
```markdown
@src/silver/transformations.py
```
Claude reads this file as part of the command. Used so commands can analyze your actual code.

**3. `$ARGUMENTS` — captures what you type after the command**
```
/add-table Suppliers
```
Inside the command file, `$ARGUMENTS` = `Suppliers`. Claude uses it to know which table to scaffold.

### All 10 Commands

#### `/run-pipeline [layer]`
**What it does:**
1. Injects live config state via `!bash` (server, database, paths, tables)
2. Pre-flight checks: all config keys present? data directories exist? JDBC jar present?
3. Runs `python src/main.py`
4. Reads latest log, shows last 20 lines
5. Reports row count for every Gold table
6. On failure: extracts ERROR lines, matches to known patterns, gives specific fix

**Example usage:**
```
/run-pipeline         → full pipeline
/run-pipeline gold    → reminder that partial layer runs need --layer flag
```

#### `/validate-schema [layer]`
Reads actual Parquet files, compares schema to expected contracts from CLAUDE.md.
Reports type mismatches as a table:
```
silver | products | unit_price | expected: float | actual: double | FAIL
```
Catches silent type drift before it causes aggregation errors in Gold.

#### `/data-quality [layer]`
Runs a comprehensive PySpark DQ script covering:
- Bronze: non-empty check
- Silver: null PK rate (must be 0%), duplicate PK detection, value range checks
- Gold: revenue reconciliation against Silver (within 0.01 tolerance)
- Referential integrity: every `customer_id` in orders exists in customers

Produces a DQ score (e.g. "11/13 checks passed — 84.6%") and writes a report to `logs/`.

#### `/add-table <TableName>`
**Safest way to add a new source table.** Reads all four source files, asks you questions
about the new table's schema and relationships, generates ALL scaffolding (SQL DDL +
Silver transform method + Gold aggregation + config.yaml entry + main.py wiring), shows
everything for review, and only writes after your confirmation.

#### `/test-layer bronze|silver|gold`
Creates `tests/conftest.py` (if missing), then generates a complete pytest file for the
requested layer with tests for type casting, null PK filter, line_total derivation, join
correctness, and aggregation math. Runs immediately and shows results.

#### `/config-audit`
Greps all `config.get()` calls across `src/`, cross-references each key against `config.yaml`,
and reports: missing keys (silent `None` returns), security issues (credentials in plaintext),
path validity, and unused keys.

#### `/git-checkpoint`
Runs `git status`, identifies only safe-to-commit files (`src/`, `tests/`, `config.yaml`,
etc.), refuses to stage `data/`, `logs/`, `.env`, reads changed files to understand what
changed, generates a semantic commit message (`feat(silver): ...`), shows a summary,
and commits after your confirmation.

#### `/spark-debug [keyword]`
Finds the latest log file, extracts ERROR/WARN/Exception lines (filtered by keyword if
provided), and matches each against 10+ known error patterns with project-specific fixes
(not generic PySpark docs — actual fixes for THIS pipeline's configuration).

#### `/lineage-report`
Reads `sql/create_schema.sql` + all three layer source files, traces every column from
SQL Server through Bronze → Silver → Gold, labels each transformation (rename, cast,
derived, aggregated, dropped), and saves a Markdown lineage table to `data/lineage_report.md`.

#### `/incremental-design`
Analyzes which tables have watermark-eligible columns (`Orders.OrderDate`), designs a
`WatermarkManager` utility class, shows the full diff needed across `ingestion.py`,
`config.yaml`, and `main.py`, and implements on your confirmation.

## 7. Agents — Specialist Sub-Brains

**File location:** `.claude/agents/*.md`

Agents are specialist definitions Claude can invoke when a task matches their domain.
Each agent file has a `description` with `<example>` blocks — Claude reads these to know
when to switch into that agent's mode.

### Agent File Format
```yaml
---
name: silver-layer-agent
description: Use this agent when the task involves Silver layer transforms...
  <example>
  user: "Write a transform method for the new Inventory table"
  assistant: "I'll use the silver-layer-agent..."
  </example>
model: inherit
color: cyan
---
```

The `model: inherit` means it uses whatever model you're running. The `color` is just
a visual identifier in Claude Code's UI.

### The 5 Agents

#### `bronze-layer-agent`
**Domain:** JDBC ingestion, SQL Server connectivity, Bronze write modes.

**Key constraint:** "NEVER add transformation logic to BronzeLayer."

Knows the exact JDBC auth bug fix, how to configure `spark.jars` for the driver jar,
how to add JDBC performance options (`fetchsize`, `numPartitions`), and when to use
`overwrite` vs `append` write modes.

**Invoked by:** `/run-pipeline`, `/add-table`, `/spark-debug`

#### `silver-layer-agent`
**Domain:** Type casting, null safety, Silver contract enforcement.

**Key constraint:** "NEVER use `select('*')`. NEVER skip the PK null filter. NEVER add
cross-table joins."

Enforces the Silver contract and the exact transform method structure. Has the full
SQL Server type → Spark type mapping table. Knows how to add StructType schema
enforcement (currently not used, but knows the pattern for when you're ready).

**Invoked by:** `/test-layer silver`, `/validate-schema`, `/add-table`, `/data-quality`

#### `gold-layer-agent`
**Domain:** Join strategies, aggregations, window functions, business metrics.

**Key constraint:** "NEVER use inner join. ALWAYS left join from order_items."

Knows to round monetary columns to 2 decimal places before saving (prevents
`49.999999998` floating-point noise). Knows to drop null groupBy keys before `groupBy()`
to prevent a "null" aggregation group. Has window function patterns for rankings and
running totals.

**Invoked by:** `/run-pipeline`, `/data-quality`, `/lineage-report`, `/add-table`

#### `test-writer-agent`
**Domain:** pytest + PySpark test infrastructure.

**Mission:** Build the empty `tests/` directory into a full test suite.

Knows the exact `conftest.py` structure with `scope="session"` SparkSession (starts once
per run, not once per test). Knows to mark SQL Server tests `@pytest.mark.integration`
so CI can skip them. Knows what to test for each layer.

**Invoked by:** `/test-layer`

#### `ci-cd-agent`
**Domain:** GitHub Actions workflows for Python + PySpark.

**Key knowledge:** Java 11 must be installed BEFORE `pip install pyspark`. Without
`SPARK_LOCAL_IP=127.0.0.1`, PySpark fails on GitHub Actions runners. Has a failure
lookup table for 8 common CI + PySpark errors.

**Invoked by:** `/git-checkpoint` (post-commit suggestion)

## 8. Hooks — Automated Scripts

**File location:** `.claude/hooks/*.py`

Hooks are Python scripts that fire automatically based on events. They receive context
as JSON on stdin and can output JSON to control Claude's behavior.

### How Hook Input/Output Works

**Input (stdin):**
```json
{
  "tool_name": "Bash",
  "tool_input": {"command": "rm -rf data/"},
  "tool_response": {"stdout": "", "stderr": "..."}
}
```

**Output for blocking (pre-hooks only):**
```json
{"decision": "block", "reason": "Destructive: rm -rf is blocked"}
```

**Output for allowing:**
```json
{"decision": "allow"}
```

Post-hooks and Stop hooks don't need to return a decision — they just run side effects.

### `pre_bash_guard.py` — Runs BEFORE every bash command

Checks the command against blocked patterns:

| Pattern | Reason |
|---|---|
| `rm -rf` | Destructive deletion |
| `git push --force` | History rewrite |
| `git reset --hard` | Local work destruction |
| `DROP DATABASE` | Catastrophic SQL Server damage |
| `> .env` | Credential file overwrite |
| `> config.yaml` | Config overwrite via redirect |

Non-blocking warnings (print to stderr, don't block):
- `git add .` — may capture `data/` or `.env`
- `git add -A` — same warning

**Critical design:** If the script crashes for any reason, it defaults to `{"decision": "allow"}`.
Hook bugs never block your session.

### `post_bash_logger.py` — Runs AFTER every bash command

Writes to two log files:

**`logs/claude_session_audit.log`** — full audit trail:
```
[2026-04-27 23:55:01] CMD: python src/main.py | OUT: Starting data pipeline...
[2026-04-27 23:56:12] CMD: git status | OUT: On branch feature/fix-jdbc-auth...
```

**`logs/pipeline_runs.log`** — pipeline execution history:
```
[2026-04-27 23:55:01] SUCCESS: python src/main.py
[2026-04-27 23:58:30] FAILURE: python src/main.py
```

The pipeline log lets you quickly check "how many times has the pipeline run and did it succeed?"
without scrolling through the full audit log.

### `session_summary.py` — Runs when Claude Code session ends

Reads both log files and prints:
```
==================================================
  Claude Code Session Summary — ClaudeDataPipeline
==================================================
  Commands run (this session) : ~14
  Python executions           : 6
  Git commands                : 3
  pytest runs                 : 2
  Pipeline runs               : 2 (1 success, 1 fail)

  Last pipeline errors:
    ! ERROR - No suitable driver found for jdbc:sqlserver
==================================================
```

### `update_claude_docs.py` — Runs when .claude/ files change

**This is the script that keeps this document up to date.**

Triggered by `PostToolUse` on `Write|Edit|MultiEdit` tools. Checks if the written file
is inside `.claude/`. If yes, scans the entire `.claude/` directory and regenerates the
[File Inventory](#file-inventory-auto-generated) section at the bottom of this document.

The inventory section is delimited by HTML comment markers:
```
<!-- CLAUDE_INVENTORY_START -->
  ... auto-generated content ...
<!-- CLAUDE_INVENTORY_END -->
```
The script finds these markers and replaces only that section, leaving the rest of
this document untouched.

**Manual regeneration:**
```bash
python .claude/hooks/update_claude_docs.py --regenerate
```

## 9. GitHub Actions Workflows

**File location:** `.github/workflows/*.yml`

Three workflows run automatically on GitHub when you push code or open pull requests.
None require a live SQL Server — all SQL-dependent tests are marked and skipped in CI.

### `pipeline-tests.yml` — Triggers on every push and PR

**Steps:**
1. Checkout code
2. **Install Java 11** (Temurin) — required before PySpark
3. Install Python 3.11
4. `pip install -r requirements.txt` — installs pyspark, pyodbc, pyyaml
5. `pip install pytest pytest-cov`
6. Run: `pytest tests/ -m "not integration" --cov=src --cov-report=xml`
7. Upload `coverage.xml` as a downloadable artifact

**Key environment variables:**
```yaml
SPARK_LOCAL_IP: 127.0.0.1   # Prevents PySpark hostname resolution failure
PYSPARK_PYTHON: python3     # Consistent Python executable
PYTHONPATH: .               # So 'from src.config...' imports resolve
```

### `code-quality.yml` — Triggers on every push and PR

Runs three linters in sequence:
- `black --check --line-length 120` — formatting (fails if code needs reformatting)
- `isort --check-only --profile black --line-length 120` — import ordering
- `flake8 --max-line-length=120 --ignore=E203,W503,E501` — style

**Why 120 chars?** PySpark method chaining naturally produces long lines. The standard
79-char limit would require breaking every Spark chain across many lines.

### `schema-validation.yml` — Triggers on PRs only (not every push)

Runs an inline Python script that loads `config.yaml` and verifies:
- All required sections exist: `sql_server`, `spark`, `pipeline`, `paths`, `tables`
- All required sub-keys are present and non-empty
- `tables.source` is a non-empty list
- Each table entry has both `name` and `bronze_table` keys

**Why PR-only?** This is a merge gate — it prevents merging a PR that breaks the pipeline
config. Running on every push would just be noise.

## 10. How to Add New Files

When you add a new file to `.claude/`, it will automatically appear in the
[File Inventory](#file-inventory-auto-generated) section below after the next Claude Code
`Write` or `Edit` operation on any `.claude/` file. To force an immediate update, run:
```bash
python .claude/hooks/update_claude_docs.py --regenerate
```

### Adding a New Slash Command

1. Create `.claude/commands/your-command-name.md`
2. Add frontmatter at the top:
   ```markdown
   ---
   description: One-line description shown in /help (under 60 chars)
   allowed-tools: Bash, Read, Glob
   argument-hint: "[optional] description of expected arguments"
   ---
   ```
3. Write the workflow instructions below the frontmatter
4. Use `!`bash cmd`` to inject live context, `@filepath` to inline files, `$ARGUMENTS` for user input
5. It immediately becomes available as `/your-command-name` in Claude Code

### Adding a New Agent

1. Create `.claude/agents/your-agent-name.md`
2. Add frontmatter:
   ```yaml
   ---
   name: your-agent-name
   description: "Use this agent when... Examples: <example>...</example>"
   model: inherit
   color: blue  # blue|cyan|green|yellow|magenta|red
   tools: Read, Bash, Edit
   ---
   ```
3. Write the agent's role, capabilities, and constraints in the body

### Adding a New Skill

1. Create `.claude/skills/your-skill-name.md`
2. Add frontmatter:
   ```yaml
   ---
   name: Your Skill Name
   description: "Use when the user asks about X or Y in this project"
   ---
   ```
3. Write domain knowledge, code patterns, and anti-patterns in the body
4. Reference the skill in `CLAUDE.md` under the Skills Reference section

### Adding a New Hook
